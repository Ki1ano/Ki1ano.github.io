SQL面试题整理

一、基础知识

1. 数据库的三范式：

（1）列不可再分。（2）行可以唯一区分，主键约束。（3）除主键外的其他列必须和主键直接相关，不能间接相关，外键约束。

2. InnoDB 与 MyISAM 的区别：

（1）事务支持（Transactions）：

InnoDB —— 支持完整的ACID属性事务（原子性、一致性、隔离性、持久性）：允许将一组SQL操作作为一个不可分割的工作单元来执行，若任一语句失败，整个事务可以回滚（ROLLBACK）—>保证数据状态一致。应用场景：需要高数据完整性的应用，如银行系统

MyISAM —— 不支持事务：每条SQL语句都是原子的，但无法将多条语句组合成一个事务并保证其原子性。如果执行过程中发生故障，只执行了部分语句，导致数据不一致。仅支持表级锁定语句（如LOCK TABLES），无法提供细粒度的事务控制。

（2）锁机制：

InnoDB —— 默认采用行级锁。当修改数据时，只锁定需要修改的特定行，其他行仍可以被其他会话并发读写（取决于事务隔离级别）—>提高了在高并发、写操作频繁的环境下的性能和可伸缩性。同时，也支持表级锁，如LOCK TABLES … WRITE。

MyISAM —— 只支持表级锁。当执行写操作时，锁定整个表，此时任何其他会话对该表的读写操作都会被阻塞，直到锁释放 —> 在大量写操作或混合读写负载的场景下，这会导致严重的性能瓶颈和并发度低下。读操作之间通常不会相互堵塞（共享锁兼容）。

（3）外键约束（Foreign Keys）：

InnoDB —— 支持外键约束。可以定义关系，并强制实施参照完整性（Referential Integrity）：当尝试插入/更新违反外键约束的数据时，InnoDB 会阻止操作并报错。级联删除和更新（ON DELETE CASCADE, ON UPDATE CASCADE）也由 InnoDB 在数据库层面处理。

MyISAM —— 不支持外键约束。表之间的关系逻辑必须在应用程序层面维护。数据库本身不会阻止插入无效的外键值 —>增加了数据不一致的风险。

（4）崩溃恢复（Crash Recovery）：

InnoDB —— 具有强大的崩溃恢复能力。使用Write-Ahead Logging (WAL) 机制：所有修改，在应用到数据文件之前，会先持久化写入到重做日志（Redo Log）文件。如果数据库意外崩溃，重启时 InnoDB 可以利用 Redo Log 自动重做崩溃前已提交、尚未写入数据文件的更改，并利用 Undo Log 回滚未提交的事务 —> 保证数据的持久性（Durability）和一致性。过程自动且可靠。

MyISAM —— 崩溃恢复能力较弱。没有事务日志（如 Redo Log）。

（5）存储结构（Storage Structure）：

InnoDB —— 聚集索引：数据文件是和索引绑在一起的（在同一文件中）。必须有主键，通过主键索引效率很高。辅助索引需要两次查询：（1）先查询到主键，（2）通过主键查询到数据。—>主键不应过大，主键太大会导致其他索引变大。通过索引查询到数据文件，性能高。

MyISAM —— 非聚集索引：数据文件是分离的（数据文件和索引文件是分离的），索引保存的是数据文件的指针。主键索引和辅助索引是独立的。二级指向，性能会下降。

（6）性能特点：

InnoDB ——

  • 高并发环境下（尤其是写操作、混合读写操作）性能通常更好，得益行级锁。处理大量写操作效率更高。

  • 复杂查询（如 JONS、子查询）在缓冲池管理下可能有良好表现。

  • 启动和关闭（涉及 Redo Log 处理）可能比 MyISAM 稍慢。

  • 由于需要维护事务、锁、MVCC等机制，内存和 CPU 开销通常更大。

MyISAM ——

  • 在纯读操作密集、且并发度不高的场景下，简单查询速度可能非常快（结构简单，缓存索引后查找直接）。
  • 执行全表扫描（无索引 或 无法使用索引）有时可能比 InnoDB 稍快（取决于具体数据和缓存）。
  • COUNT(*)操作在没有 WHERE 子句时极快（ MyISAM 存储了总行数）。
  • 在写操作频繁或高并发读写环境下，性能急剧下降，主要受表级锁限制。
  • 内存和 CPU 开销通常较低（机制简单）。

（7）全图索引：

InnoDB —— 从 MySQL5.6 开始支持内置的全文索引。InnoDB 的 FTS 在功能和性能上更优。

MyISAM —— 很早就支持内置的全文索引。

总结：除 明确经过验证的、MyISAM 能带来显著优势的特定只读场景，否则 InnoDB 是 MySQL 存储引擎的绝对首选和标准。

MyISAM —— 崩溃恢复能力较弱。没有事务日志（如 Redo Log）。
